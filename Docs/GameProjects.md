# Game Projects (descriptor-driven build system)

Zenith games follow the Unity/Unreal model: create a game with **zero build-script
edits**, and open a solution containing **only the engine and the one game** you are
working on. Sharpmake (in-repo) is still the generator — CMake is not used.

## TL;DR

```
zenith new <Name>          # scaffold a game, regen, open its solution
zenith build <Name>        # build the game (its per-game sln, /t:<Game>)
zenith run   <Name>        # launch the newest built exe
zenith test  <Name|all>    # run the game's automated tests (THE test entry point)
zenith open  <Name>        # regen + open the game's solution in Visual Studio
zenith list                # list games + built configs
zenith regen [--check]     # regenerate every solution (--check: staleness report)
zenith clean [<Name>]      # kill hanging build processes (+ wipe output/obj)
zenith package <Name>      # stage a relocatable build into dist/
zenith hub                 # launch the Unity-Hub-style GUI launcher
zenith selftest            # run the PowerShell test suites
```

`zenith` is `C:\dev\Zenith\zenith.bat` (→ `Tools/zenith.ps1` → `Tools/ZenithCli`).

## The descriptor — `Games/<Name>/<Name>.zproj`

Each game is fully described by a small JSON file checked in at the game root:

```json
{
    "schemaVersion": 1,
    "name": "Sokoban",
    "android": true,
    "extraDefines": [],
    "extraSharpmakeProjects": []
}
```

| Key | Meaning |
|-----|---------|
| `schemaVersion` (required) | Must be `1`. A higher value is a hard error (forward descriptors must not be mis-generated by an old tool). |
| `name` (required) | `^[A-Z][A-Za-z0-9]{0,63}$` (PascalCase, ASCII, ≤ 64 chars) **and** must equal both the folder name and the file basename (triple redundancy catches copy-a-game mistakes). |
| `android` (default `false`) | Emits AGDE targets + `<name>_agde.sln` and drives `deploy_android` discovery. Requires a `Games/<Name>/Android/` Gradle tree. |
| `extraDefines` (default `[]`) | Extra preprocessor defines for the game. |
| `extraSharpmakeProjects` (default `[]`) | Extra Sharpmake project **type names** to include in this game's solution (e.g. TilePuzzle carries `TilePuzzleLevelGenProject` + `TilePuzzleRegistryViewerProject` so its offline tools live in *its* sln, keeping the engine sln game-free). |

Unknown JSON keys are a **hard error** (a typo net — `andriod` or `extraDefine` must
not silently no-op).

**Android note:** `CityBuilder` and `Test` are `android:false` (they have no `Android/`
Gradle tree). The other 10 games are `android:true`.

## Generation — one command, one Sharpmake run

`Build/regen.ps1` (canonical; `zenith regen` forwards to it) does, in order:

1. **Refuse to run from a linked git worktree** (Sharpmake-in-worktree generates
   paths that resolve against the wrong tree).
2. **Validate** every `Games/*/*.zproj` (all errors collected, not fail-fast).
3. **Codegen** `Build/Sharpmake_GameInstances.generated.cs` from the descriptors —
   per-game `[Sharpmake.Generate]` `<Name>GameProject` + `<Name>GameSolution` shells
   (reusing today's class names → zero GUID churn) plus a `GeneratedGameManifest`
   (descriptor path + SHA256 per game).
4. Build the `/sources` list from a `Build/Sharpmake_*.cs` glob (no stale hand-lists)
   and **run Sharpmake once** → engine sln + all per-game slns in one consistent snapshot.
5. Fix up AGDE vcxproj (`fix_agde_vcxproj.ps1` — c++ standard + UBSan).
6. Delete the obsolete monolithic `zenith_win64.sln` / `zenith_agde.sln`.

`-UseDotnet` runs Sharpmake via `dotnet exec …Sharpmake.Application.dll` (CI uses this).

### Manifest guard

`Build/Sharpmake_Solutions.cs` (`[Sharpmake.Main]`) re-hashes the on-disk `.zproj`
files and compares against `GeneratedGameManifest`. Any drift (a descriptor added /
changed / removed since the last codegen) throws **"run Build/regen.ps1"**. Running
Sharpmake without the generated file fails to compile (missing manifest type) — a
deliberate failsafe. The C# side never parses JSON structurally; the hash is the gate.

## Solution inventory

Sharpmake lowercases solution filenames.

| Solution | Location | Contents |
|----------|----------|----------|
| `<name>_win64.sln` | `Games/<Name>/` | ZenithBase/ECS/Physics/AI + Zenith aggregate + FluxCompiler (Vulkan) + font libs (tools) + **the one game** + its `extraSharpmakeProjects`. No Sentinels, no other games. |
| `<name>_agde.sln` | `Games/<Name>/` | Same, Android — only for `android:true` games (10 of 12). |
| `zenith_engine_win64.sln` | `Build/` | Engine libs + Sentinels (`_False`) + FluxCompiler + font libs + **ZenithHub**. **Zero games, ever.** Engine CI gates + pure engine work. |

The game **vcxproj** files stay under `Games/<Name>/Build/` (so output dirs and config
names are unchanged); only the `.sln` moved to the game root.

Config names (`Vulkan_vs2022_Debug_Win64_True` …) and output dirs
(`Games/<Name>/Build/output/<platform>/<config>/`) are **unchanged** from before.

## Building — always `/t:<Game>`, never the whole sln

```
msbuild Games\Sokoban\sokoban_win64.sln /t:Sokoban /p:Configuration=Vulkan_vs2022_Debug_Win64_True /p:Platform=x64
msbuild Build\zenith_engine_win64.sln   /t:Zenith  /p:Configuration=Vulkan_vs2022_Debug_Win64_True /p:Platform=x64
```

A whole-solution build pulls in the aux tools (FluxCompiler / TilePuzzle tools) that
are **pre-existing-red in `ToolsEnabled=True`** — out of scope here. `zenith build`
and `zenith build engine` always use `/t:`.

## Git tracking

ALL Sharpmake outputs are **gitignored** — every `.sln` (`Games/*/*.sln`,
`Build/zenith_engine_win64.sln`), every `.vcxproj`/`.filters`/`.user` (engine, game,
and tool projects alike), and `Build/Sharpmake_GameInstances.generated.cs` — all
regenerated by `regen`. Nothing generated is ever committed (regenerate-first policy;
the historical partial tracking of game vcxprojs was removed 2026-07).

**Troubleshooting:** `.vcxproj`/`.sln` missing after a fresh clone or branch switch →
run `Build/regen.ps1` (or `zenith regen`). `zenith regen --check` reports whether the
on-disk generated files are stale relative to the descriptors without regenerating.

## CI

Every workflow runs `pwsh Build/regen.ps1 -UseDotnet` first, then builds via `/t:`:

| Workflow | Regen + build |
|----------|---------------|
| `cb-tests.yml` | `msbuild Games\CityBuilder\citybuilder_win64.sln /t:CityBuilder` (Vulkan `_True` + D3D12 `_False`) |
| `dp-tests.yml` | `msbuild Games\DevilsPlayground\devilsplayground_win64.sln /t:DevilsPlayground` (same) |
| `shader-validation.yml` | `msbuild Build\zenith_engine_win64.sln /t:FluxCompiler` (first consumer + permanent gate of the engine sln) |
| `scaffold-smoke.yml` | path-filtered; `Tools/test_scaffold.ps1` (`zenith new` → build → boot → teardown) |

## The hub — `zenith hub`

A standalone ImGui + Win32 + D3D11 launcher (`ZenithHub/`, engine-sln member). Lists
every game (name / android / built configs / newest build) with per-row Open + Run, a
New Game modal (live name validation, template dropdown, Android checkbox), and a Regen
button. It **shells the CLI for every mutation** — zero reimplemented logic.
`zenithhub.exe --selftest` runs its descriptor-parse + shared-name-vector + scan tests.

## Templates — `Build/Templates/<T>/`

`zenith new <Name>` expands a template into `Games/<Name>/`. Tokens: `{{GAME_NAME}}` /
`{{GAME_NAME_LOWER}}` / `{{GAME_NAME_UPPER}}` in content, `__GAME_NAME__` in filenames,
`.in` suffix stripped on expansion (expansion hard-fails if any `{{` survives). The
`NewGame` template ships a bobbing-lit-cube game.

**First-run scene caveat:** `Assets/Scenes/Main.zscen` is **boot-authored**, not checked
in (a `.zscen` would rot with engine churn). `Project_RegisterEditorAutomationSteps`
(tools-only) authors + `AddStep_SaveScene`s it, so the **first build+run must be a
`*_True` config** to bake the scene; thereafter `_False`/Android load it.

## Troubleshooting

- **"…generated.cs is STALE… run Build/regen.ps1"** — you edited a `.zproj` (or the
  generated file is missing/old). Run `Build/regen.ps1` (or `zenith regen`).
- **Stale `zenith_win64.sln`** — the monolithic slns are gone; `regen` deletes them.
  Use the per-game sln or `zenith open <Name>`.
- **`MSB8013` / "does not contain a project"** — you built a bare `.vcxproj`. Build the
  `.sln` with `/t:<Game>`.
- **Refuses to regen "from a linked git worktree"** — run from the main checkout.

## Rollback

`git revert` the cutover commit, run `Build/regen.ps1`, and delete any orphaned
per-game / engine slns. Sharpmake GUIDs are deterministic, so regen restores a
consistent state.
